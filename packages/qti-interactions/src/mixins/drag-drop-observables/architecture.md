# Drag-Drop Observables Architecture

## Purpose

This document is the authoritative architecture and UX contract for the drag-drop observables mixin library.
It exists to keep interaction behavior consistent, auditable, and validation-safe across all implementations.

## Scope

In scope:
- `drag-drop-core.mixin.ts`
- `drag-drop-slotted.mixin.ts`
- `drag-drop-sortable.mixin.ts`
- `drag-drop-slotted-sortable.mixin.ts`
- `utils/*`, `strategies/*`
- every interaction component that composes these mixins

Out of scope:
- non-mixin interactions (documented separately)
- visual theming details that do not affect behavior

## Composition Model

```text
DragDropCoreMixin
  -> DragDropSlottedMixin
    -> DragDropSlottedSortableMixin
  -> DragDropSortableMixin
```

Responsibilities:
- `Core`: pointer/mouse/touch/keyboard streams, drag clone lifecycle, collision targeting, active/hover states.
- `Slotted`: inventory-to-dropzone behavior, association counting, max/min association validation, match-max gating.
- `Sortable`: placeholder-driven in-container reordering.
- `SlottedSortable`: slot-origin reordering/swap policy layered over slotted behavior.

## Canonical UX Patterns

All mixin-based interactions must follow these UX patterns.

1. Direct manipulation parity
- Pointer, touch, and keyboard flows must commit equivalent domain outcomes.

2. One active drag at a time
- No duplicate drag starts from mixed input streams.

3. Clear drop affordances
- Valid droppables are enabled/active during drag.
- Current target is singular and visible.

4. Deterministic drop handling
- Drop either commits to exactly one target or rolls back deterministically.

5. Capacity-aware disablement
- Dropzones are disabled when global max-associations is reached or local `match-max` is reached.
- Source droppable return remains allowed when marked as drag source.

6. Layout stability
- Drag should not cause avoidable content reflow during hold/move.
- Source-removal behavior must preserve spatial continuity where possible.

## Single-Occupancy Swap Pattern (Required)

For droppables with effective capacity `1` (`match-max=1`):
- Dropping a draggable onto an occupied droppable must swap/move deterministically.
- The existing occupant is restored to inventory (or equivalent source location) and the dragged item takes its place.
- This applies both for inventory-origin drags and slot-origin drags.

For droppables with capacity `>1` or unlimited (`match-max=0`):
- occupied target accepts append behavior until capacity is reached.
- once capacity is reached, additional append attempts are blocked.

This behavior is implemented in `dropDraggableInDroppable`:
- single-capacity target path removes existing item and restores inventory instance
- multi-capacity target path appends additional draggable clone until full

## Drag Lifecycle Contract

1. `cacheInteractiveElements`
- collect draggables, droppables, inventory containers from light and shadow scopes
- apply baseline draggable attributes/styles (`qti-draggable`, cursor, `tabindex`, touch-action)

2. `initiateDrag`
- detect origin (inventory vs droppable)
- create visual clone
- mark source droppable (`data-drag-source`) when drag starts from a droppable

3. `handleDragMove`
- move clone
- compute collision target
- update hover states
- update sortable placeholder where applicable

4. `handleDragEnd`
- resolve drop target
- commit or rollback based on `allowDrop` and disabled/capacity rules
- remove clone, clear temporary states, persist response

5. cleanup/reset
- remove placeholder/clone artifacts and subscriptions
- restore hidden inventory items and clear transient attributes

## Invariants and Guardrails

Behavior invariants:
- at most one active drag per interaction instance
- drag clone is never draggable/focusable
- no stale `data-drag-source` after drag end
- no hidden inventory source left after completion
- response must be derivable from final DOM state

Implementation guardrails:
- avoid per-interaction branching in `Core` where a hook can be used
- keep capacity logic centralized in `Slotted` + utility functions
- keep theme-only styling out of drop correctness logic

## Known Risk Areas

1. Inline reflow during slot-origin drag
- source element removal can toggle `:empty` styles and shift content
- mitigation: reserve source geometry during drag (placeholder or hidden-in-place strategy)

2. Cross-input event races
- pointer/mouse compatibility events can duplicate initiation without guard windows

3. Temporary droppable enablement in sortable slot moves
- any temporary enable must always be reverted on all termination paths

## Testing Contract

Every behavior change in this library should include coverage for:
- drag start deduplication across input streams
- source/clone/placeholder cleanup on valid and invalid drops
- single-capacity swap behavior
- multi-capacity append and full-capacity blocking
- max/min association validation messages and validity state
- shadow-root and light-DOM droppable paths
- layout-stability checks for inline content interactions
